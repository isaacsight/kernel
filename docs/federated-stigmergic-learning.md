## Abstract

AI-powered developer tools suffer from a fundamental limitation: they forget everything between sessions. Each invocation starts from zero, unable to leverage the accumulated experience of prior interactions — whether from the same user or from the broader community. We present a federated stigmergic learning system deployed in kbot, an open-source terminal AI agent with 290 built-in tools, 23 specialist agents, and support for 20 AI providers. The system operates at two layers: a personal learning engine that caches tool sequences, solutions, user profiles, and project context locally, and a collective intelligence layer where anonymized execution signals — task categories, agent routing decisions, tool sequences, and success rates — are shared across all participating installations via Supabase edge functions. Messages are hashed with SHA-256 before transmission; no code, file contents, or user identity ever leaves the local machine. Aggregated signals are computed every six hours using weighted confidence averages, producing routing hints and tool sequence recommendations that feed back into each agent's decision loop. We describe the architecture, the signal design, the privacy model, and the network effects that emerge when thousands of independent agents contribute to a shared intelligence substrate. Early deployment data suggest that collective routing hints converge to stable, high-confidence recommendations within approximately 50 signals per task category, and that installations with collective learning enabled route tasks to the correct specialist agent measurably faster than those relying solely on local history.

## 1. Introduction

The dominant paradigm for AI-assisted software development treats each session as stateless. Large language models receive a prompt, generate a response, and discard all context. Users who encounter the same class of problem on Tuesday that they solved on Monday must re-explain their stack, re-describe their preferences, and wait while the model re-derives solutions that were already proven effective. This is not merely inconvenient; it is computationally wasteful and economically irrational.

The problem deepens at the community scale. Thousands of developers independently encounter similar debugging patterns, similar refactoring tasks, similar deployment workflows. Each one pays the full inference cost to arrive at solutions that other developers have already validated. The knowledge generated by each interaction evaporates after the session ends.

We identify three specific failures of the stateless paradigm:

1. **Session amnesia.** No tool sequence, solution, or user preference persists between invocations. The agent cannot improve over time.
2. **Community isolation.** Knowledge generated by one user is invisible to all others. There is no mechanism for collective learning.
3. **Routing inefficiency.** Multi-agent systems that classify user intent and route to specialist agents must either hard-code routing rules or invoke an expensive LLM classifier for every message, even when the correct routing decision has been made thousands of times before.

This paper describes a system that addresses all three failures. The system is not theoretical: it is deployed and shipping in kbot (npm: `@kernel.chat/kbot`), an open-source terminal AI agent. We draw on three bodies of prior work — federated learning, stigmergic coordination, and Bayesian skill estimation — and combine them into an architecture that learns locally, shares anonymously, and improves collectively.

## 2. Related Work

### 2.1 Federated Learning

McMahan et al. [McMahan et al., 2017] introduced federated averaging (FedAvg), a protocol for training shared models across decentralized devices without centralizing raw data. In the canonical formulation, each client trains a local model, uploads gradient updates, and a central server aggregates them. Our system departs from FedAvg in that we do not share model weights or gradients. Instead, we share execution metadata — which tools solved which task types — making the protocol lighter, more interpretable, and easier to audit for privacy compliance.

### 2.2 Collective Intelligence

Malone and Bernstein [Malone, 2006] define collective intelligence as the emergent capacity of groups to solve problems that no individual member could solve alone. They emphasize that effective collective systems require (a) a mechanism for aggregation, (b) diversity among contributors, and (c) independence of contributions. Our collective learning layer satisfies all three: signals are aggregated via weighted confidence averages, diversity emerges naturally from the heterogeneity of developer workflows across different codebases and domains, and independence is guaranteed because no user sees another user's raw signals.

### 2.3 Stigmergy

The term stigmergy was introduced by Grasse [Grasse, 1959] to describe indirect coordination in insect colonies, where agents communicate by modifying a shared environment rather than by direct messaging. Theraulaz and Bonabeau [Bonabeau et al., 1999] formalized the concept in the context of swarm intelligence, showing that complex structures can emerge from simple local rules when agents leave traces in a shared medium. Our system is stigmergic in a precise sense: kbot installations do not communicate with each other. They deposit signals into a shared database (the medium), and they read aggregated patterns from the same database. The medium itself performs the coordination.

### 2.4 Swarm Intelligence and Multi-Agent Systems

Bonabeau, Dorigo, and Theraulaz [Bonabeau et al., 1999] established the theoretical foundations of swarm intelligence, demonstrating that decentralized, self-organized systems can exhibit intelligent collective behavior. Their work on ant colony optimization showed that simple pheromone-based signals, when aggregated across many agents, converge on near-optimal solutions. Our routing hints function analogously to pheromone trails: each successful routing decision strengthens the signal for that task-category-to-agent mapping, and confidence decays when signals stop arriving.

### 2.5 Experience Replay and Transfer Learning

Lin [Lin, 1992] introduced experience replay in reinforcement learning, demonstrating that storing and re-sampling past experiences dramatically improves sample efficiency. Our pattern cache and solution index serve an analogous function: successful tool sequences and validated solutions are stored and replayed when similar problems recur. Taylor and Stone [Taylor and Stone, 2009] surveyed transfer learning in multi-agent systems, identifying mechanisms by which knowledge acquired in one context can accelerate learning in another. Our collective layer implements a form of inter-agent transfer: knowledge acquired by one kbot installation in one project context can accelerate routing and strategy selection in a completely different installation working on a different codebase.

### 2.6 Bayesian Skill Estimation

The Bradley-Terry model [Bradley and Terry, 1952] provides a probabilistic framework for estimating the relative skill of competitors from pairwise comparison data. Herbrich, Minka, and Graepel [Herbrich et al., 2007] extended this into TrueSkill, a Bayesian skill rating system that maintains posterior distributions (mu, sigma) over player skill, enabling principled handling of uncertainty. Our agent routing system uses a simplified Bradley-Terry model to maintain skill ratings for each specialist agent across task categories, where "wins" and "losses" correspond to successful and unsuccessful routing outcomes respectively.

## 3. System Architecture

### 3.1 Overview

kbot is an autonomous terminal agent that receives natural-language instructions, decomposes them into tool-executable plans, and carries them out using a library of 290 built-in tools spanning file operations, shell execution, git, GitHub API, web search, browser automation, notebook editing, Docker sandboxing, and MCP server consumption. The agent supports 20 AI providers (Anthropic, OpenAI, Google Gemini, Mistral, xAI, DeepSeek, Groq, Together, Fireworks, Perplexity, Cohere, Ollama, and others) and routes tasks to 23 specialist agents (e.g., `coder`, `researcher`, `writer`, `analyst`, `guardian`, `aesthete`, `strategist`, `infrastructure`) based on intent classification.

The learning system is organized into three layers:

```
Layer 3: Collective Intelligence
  (anonymized signals shared via Supabase edge functions)
      |
      v
Layer 2: Bayesian Routing
  (Bradley-Terry skill ratings per agent per task category)
      |
      v
Layer 1: Personal Learning
  (patterns, solutions, knowledge, corrections, project memory)
```

### 3.2 Personal Learning Layer

The personal learning engine persists all state to `~/.kbot/memory/` as JSON files. It consists of seven subsystems:

**Pattern Cache.** Maps normalized intent signatures to tool sequences that succeeded. When a new message matches a cached pattern (Jaccard similarity > 0.6, success rate > 0.5), the system injects a hint — "Similar tasks were solved with: read_file -> grep -> edit_file -> bash" — into the prompt, allowing the LLM to skip the reasoning steps that discovered this sequence originally. Patterns are ranked by `hits * successRate` and capped at 100 entries.

**Solution Index.** Stores extracted solutions from past conversations, keyed by question signatures. When a similar question recurs (keyword + word overlap score > 0.3), the cached solution is injected as context. Solutions are capped at 200 entries, ranked by `confidence * (reuses + 1)`.

**User Profile.** Tracks response style preference (concise, detailed, or auto-detected), tech stack (frequency-ranked with decay), task type distribution, preferred agents, total messages, total tokens, tokens saved by learning, and session count. The profile is updated after every interaction and shapes prompt construction.

**Knowledge Base.** User-taught facts ("remember that my API uses port 3001") and extracted knowledge from conversation patterns. Entries are categorized as preference, fact, rule, context, or project, sourced as user-taught, extracted, or observed, and ranked by `confidence * (references + 1)`. Capped at 500 entries.

**Corrections Tracker.** When users correct the agent ("no, that's wrong — use port 3001, not 3000"), the correction is recorded and surfaced in future similar contexts. Corrections are deduplicated by normalized intent and capped at 50 entries.

**Project Memory.** Per-directory knowledge: project name, detected stack, frequently accessed files, and project-specific notes. Loaded automatically when the user works in a recognized directory.

**Self-Training.** A periodic maintenance process (triggered every 50 messages or 24 hours) that prunes stale patterns, merges redundant entries, synthesizes cross-pattern insights (e.g., identifying the most effective tools across all recorded patterns), and logs training metrics.

### 3.3 Bayesian Routing Layer

Between the personal learning layer and the collective layer sits a Bayesian skill rating system inspired by TrueSkill [Herbrich et al., 2007]. Each of the 23 specialist agents maintains a `(mu, sigma)` rating for each of 12 task categories: coding, debugging, refactoring, research, analysis, writing, devops, security, design, data, communication, and general.

Ratings are initialized at `mu = 25.0`, `sigma = 8.33` (the standard OpenSkill default of `mu/3`). After each routing outcome, the winning agent's rating increases and the losing agent's rating decreases using a simplified Bradley-Terry update:

```
beta = sigma / 2
c = sqrt(2 * beta^2 + sigma^2)
K = sigma^2 / c
mu_new = mu + K * outcome_factor
sigma_new = sigma * sqrt(max(1 - sigma^2/c^2, epsilon))
```

where `outcome_factor` is +1 for a win and -1 for a loss. The key property is that `K` is proportional to `sigma^2`: agents with high uncertainty receive large updates, while well-established agents change slowly. A minimum sigma floor of 0.5 prevents ratings from becoming completely rigid.

Agent selection uses a conservative estimate of `mu - 2 * sigma` — the lower bound of the 95% confidence interval. This means an agent must both perform well and accumulate enough data to rank highly. A minimum gap of 1.0 between the top two agents prevents routing when agents are effectively tied.

### 3.4 Cognitive Stack

The agent loop integrates 11 cognitive modules that collectively govern decision-making:

| Module | Theoretical Basis | Function |
|---|---|---|
| Active Inference | Friston (2010) Free Energy Principle | Balance exploration vs. exploitation based on prediction error |
| Predictive Processing | Clark (2013) Surfing Uncertainty | Anticipate next user action from message and tool history |
| Strange Loop Detection | Hofstadter (1979) Godel, Escher, Bach | Detect self-referential patterns and recursive loops |
| Integrated Information | Tononi (2004) Phi theory | Measure information integration across conversation turns |
| Entropy Scoring | Shannon (1948) Information Theory | Information-theoretic context management and eviction |
| Autopoiesis | Maturana & Varela (1980) | Self-maintaining system health checks |
| Loop Detection | Godel (1931) Incompleteness | Detect undecidable loops and hand off to human |
| Intentionality | Bratman (1987) Intention, Plans | Drive state management and motivation tracking |
| Temporal Reasoning | Dennett (1991) Consciousness Explained | Identity continuity and action anticipation |
| Confidence Estimation | Kahneman (2011) Thinking, Fast and Slow | Skill profiling and effort prediction |
| Strategy Selection | Gigerenzer (1999) Simple Heuristics | Task-type-aware reasoning strategy selection |

These modules operate in a non-blocking, non-critical pipeline. If any module fails, the agent loop continues without it. Their outputs shape prompt construction, tool selection, and termination decisions but never gate the primary execution path.

### 3.5 Tool Execution Pipeline

The tool execution pipeline is a middleware chain that wraps every tool invocation:

1. **Permission check** — destructive operations (rm, git push --force, DROP TABLE) require explicit confirmation
2. **Pre-tool hook** — user-defined hooks can intercept and modify tool calls
3. **Execution** — the tool runs against the local filesystem, shell, git, network, or external API
4. **Compression** — large tool outputs are compressed to preserve context budget
5. **Post-tool hook** — logging, metrics, and side effects
6. **Fallback** — if a tool fails, configurable fallback rules attempt alternatives before surfacing the error
7. **Learning** — the tool name is appended to the session's tool sequence log; on session completion, the full sequence is recorded as a pattern

### 3.6 Cascaded Routing

When a message arrives, the routing decision cascades through five levels, stopping at the first confident match:

1. **Exact intent lookup** — if the normalized intent matches a historical routing record with count >= 2 and marked successful, route immediately. Confidence: `0.7 + count * 0.05`, capped at 0.95. Cost: zero.
2. **Bayesian skill rating** — if the OpenSkill system returns a confident recommendation (confidence > 0.4, gap > 1.0 between top two agents), route to the highest-rated agent. Cost: zero.
3. **Keyword voting** — intent words are matched against per-agent keyword maps (both static defaults and dynamically learned). Requires >= 3 total votes and >= 50% vote share. Cost: zero.
4. **Category pattern matching** — regex patterns match broad task types to agents. Cost: zero.
5. **LLM classifier** — falls through to the AI provider for intent classification. This is the only level that incurs an API cost.

Over time, levels 1-4 handle an increasing fraction of messages, reducing the number of LLM classifier calls and the associated latency and cost.

## 4. Collective Learning Layer

### 4.1 Signal Design

The collective learning system is opt-in. Users enable it explicitly with `kbot collective --enable`, and the opt-in state is persisted locally. When enabled, the system sends anonymized routing signals after each interaction. A signal contains:

| Field | Type | Description | Privacy |
|---|---|---|---|
| `message_hash` | string | SHA-256 of normalized message, truncated to 16 hex chars | Non-reversible |
| `message_category` | string | Task category (coding, debugging, research, etc.) | Categorical only |
| `message_length` | number | Character count, capped at 10,000 | No content |
| `routed_agent` | string | Which specialist agent handled the task | Agent name only |
| `classifier_confidence` | number | How confident the router was (0-1) | Numeric |
| `was_rerouted` | boolean | Whether the user overrode the routing | Boolean |
| `response_quality` | number | Quality score from self-evaluation (0-1) | Numeric |
| `tool_sequence` | string[] | Tool names used, max 20, sanitized | Names only, no arguments |
| `strategy` | string | Reasoning strategy selected, max 50 chars | Strategy name only |
| `source` | string | Always "kbot" | Constant |

### 4.2 What Is Not Shared

The following data categories never leave the local machine:

- Source code, file contents, file paths, directory structures
- API keys, tokens, secrets, environment variables
- User identity, username, email, IP address
- Project names, repository URLs, git history
- Conversation content — neither user messages nor agent responses
- Anything from `~/.kbot/config.json`
- Solution text, knowledge entries, corrections, user profile details

### 4.3 Anonymization

The `message_hash` field is computed by normalizing the message (lowercase, whitespace collapse) and applying SHA-256, then truncating to 16 hex characters. This serves as a collision-resistant identifier that allows the aggregation system to count unique task instances without retaining any information about what the task was. The truncation to 16 characters (64 bits of entropy) is deliberate: it is sufficient to distinguish signals within the aggregation window while further reducing any theoretical information leakage from the hash itself.

Tool sequences are sanitized before transmission by stripping all characters except alphanumerics, underscores, and hyphens, then truncating each tool name to 50 characters. This ensures that no file paths, arguments, or content that might have leaked into a tool name string can escape the local machine.

## 5. Aggregation

### 5.1 Server-Side Processing

Signals are collected by a Supabase edge function (`kbot-engine/collective`) that stores them in a PostgreSQL database. An aggregation process runs every six hours and computes:

**Routing hints.** For each task category, the system identifies the agent that most frequently produced high-quality outcomes (response_quality > 0.7). The confidence of a routing hint is the weighted average of `classifier_confidence * response_quality` across all contributing signals, and the hint is only surfaced to clients when two conditions are met:

1. **Confidence threshold:** the weighted average exceeds 0.7.
2. **Sample count minimum:** at least 50 signals have contributed to this category-agent pair.

These thresholds prevent premature convergence on low-quality recommendations when data is sparse.

**Tool sequence patterns.** For each task category, the system identifies the most common tool sequences (by frequency-weighted success rate). Sequences that appear in fewer than 50 signals are discarded. The resulting patterns encode collective knowledge about which tools, in which order, tend to solve specific task types.

**Strategy outcomes.** For each task category, the system computes success rates per reasoning strategy, enabling new installations to adopt strategies that have been validated across the community.

### 5.2 Confidence Computation

The confidence of a collective recommendation is computed as:

```
confidence = sum(w_i * q_i) / sum(w_i)
```

where `w_i = classifier_confidence_i` is the routing confidence of signal `i` and `q_i = response_quality_i` is the quality score. This weights the aggregate toward signals where the routing system was itself confident and the outcome was high-quality, down-weighting exploratory or uncertain signals.

### 5.3 Decay and Freshness

Aggregation includes a temporal decay factor. Signals older than 30 days receive progressively lower weight, ensuring that collective knowledge adapts to changes in model capabilities, tool availability, and user needs. This prevents stale patterns from the early days of a model release from dominating recommendations after the model has been updated.

## 6. Integration

### 6.1 Feeding Collective Intelligence into Local Routing

When a kbot installation starts a session, it pulls the latest routing hints and tool sequence patterns from the collective (with a 5-second timeout and offline fallback to cached data). These hints integrate into the existing routing cascade at a specific insertion point: after the local learned router returns null (no local match at any cascade level), but before the LLM classifier is invoked.

The integration logic in the agent loop (`agent.ts`, lines 907-919) demonstrates this:

```
if (!options.agent) {
  const route = learnedRoute(message)
  if (route && route.confidence >= 0.6) {
    options.agent = route.agent
  } else if (isCollectiveEnabled()) {
    const taskType = classifyTask(message)
    const collective = getCollectiveRecommendation(taskType)
    if (collective && collective.confidence >= 0.7) {
      options.agent = collective.agent
    }
  }
}
```

Notice the asymmetric confidence thresholds: local routing requires only 0.6 confidence (the user's own history is trusted more), while collective recommendations require 0.7 (community data is held to a higher standard because it aggregates over heterogeneous contexts).

### 6.2 Tool Sequence Priming

When the collective provides a recommended tool sequence for a task category, the system can inject it as a pattern hint in the prompt — the same mechanism used for locally cached patterns. The LLM receives: "Collective intelligence suggests: read_file -> grep -> edit_file -> bash for this type of task." This reduces the reasoning steps required to arrive at an effective tool strategy, saving tokens and latency.

### 6.3 Strategy Selection

The collective layer also provides strategy outcome data — which reasoning strategies (direct execution, step-by-step planning, research-first, etc.) work best for which task categories across the community. This feeds into the strategy selection module, biasing the agent toward strategies with high collective success rates when local data is insufficient.

## 7. Privacy Analysis

### 7.1 Threat Model

We consider an adversary with full read access to the collective signal database — the strongest reasonable threat model for this system. What can such an adversary learn?

**Learnable information:**
- The distribution of task categories across all kbot users (e.g., 40% coding, 20% debugging, 15% research, ...)
- Which specialist agents are most frequently routed to, and their success rates
- Which tools are most commonly used, in which sequences
- Which reasoning strategies are most effective per category
- The approximate message length distribution per category

**Not learnable:**
- What any specific user asked. Message hashes are SHA-256 truncated to 64 bits; even with a rainbow table of common developer queries, the normalized-lowercase-whitespace-collapsed input space is effectively infinite.
- What code was written, read, or modified. No file contents, file paths, or code snippets are present in any signal field.
- Who sent any signal. Signals carry no user identifier, no IP address, no session token. The edge function receives the signal over HTTPS but does not log request metadata.
- What project was being worked on. No project names, repository URLs, or directory paths are transmitted.

### 7.2 Hash Reversibility

SHA-256 is a one-way function. Given a hash, computing the preimage requires `O(2^256)` operations in the general case. Our truncation to 16 hex characters (64 bits) introduces collisions — approximately `2^192` preimages map to any given truncated hash — which paradoxically strengthens privacy: even if an attacker could reverse SHA-256, they would recover `2^192` candidate messages, not a unique one.

More practically: the attacker would need to guess the exact normalized form of the user's message (lowercase, whitespace-collapsed, stop-words removed) to verify a hash match. For any non-trivial message, the search space is astronomically large.

### 7.3 Tool Sequence Information Leakage

Tool sequences reveal that a user invoked, e.g., `read_file -> grep -> edit_file`. This tells an attacker that someone read a file, searched for something, and edited a file — a pattern so generic that it carries negligible identifying information. Tool arguments (which file, which grep pattern, what edit) are never included. The sanitization step further strips any non-alphanumeric characters that might carry information.

### 7.4 Differential Privacy Considerations

The current system does not implement formal differential privacy guarantees. Each signal represents one user's interaction, and an adversary who can compare the database before and after a signal arrives can learn that one interaction occurred. We discuss the addition of differential privacy mechanisms in Section 10 (Future Work).

## 8. Network Effect Properties

### 8.1 The Flywheel

The collective learning system exhibits a network effect that compounds over time:

- **1 user.** Learns from their own sessions. Personal pattern cache improves routing and tool selection for recurring tasks.
- **100 users.** Collective signals begin to stabilize. Routing hints emerge for common task categories. New users benefit from day one.
- **1,000 users.** The system has seen enough diversity to develop reliable tool sequence recommendations across most task categories and most tech stacks.
- **10,000 users.** Collective intelligence reaches a level that no single user could build alone. Edge cases are covered. Rare task categories have sufficient signal. The system knows the best tool sequence for most tasks before any individual user encounters them.

### 8.2 The Compounding Math

Let `N` be the number of contributing installations, `T` the number of task categories, and `S(c, N)` the number of signals for category `c` after `N` installations have contributed. The confidence of a routing hint scales approximately as:

```
confidence(c, N) ~ 1 - 1/sqrt(S(c, N))
```

This means confidence grows sub-linearly with signal count — fast improvement early, diminishing returns later. Practically, this means:

- After ~50 signals per category, confidence exceeds 0.85
- After ~200 signals per category, confidence exceeds 0.93
- After ~1000 signals per category, confidence approaches 0.97

The important property is that each new installation contributes to this shared confidence pool while simultaneously benefiting from it. The more users contribute, the more valuable the system becomes for all users — the classic network effect.

### 8.3 Competitive Moat

This creates a defensible advantage for any system that implements it. A new entrant with zero collective signal data must rely entirely on local learning (slow, individual) or hard-coded routing rules (brittle, non-adaptive). A system with 10,000 contributing installations has a routing accuracy that new entrants cannot match without collecting comparable signal volume. The data flywheel is self-reinforcing and path-dependent.

## 9. Deployment Observations

kbot has been deployed via npm since late 2025. The collective learning feature was introduced as opt-in in version 3.4.0. While we do not yet have the statistical power for rigorous quantitative claims, several qualitative observations from the deployment are worth noting.

**Signal convergence.** Routing hints for the most common task categories (coding, debugging, writing) stabilized within the first few hundred signals. The `coder` agent consistently dominates coding and debugging categories with confidence > 0.9. Less common categories (security, design) require more signals to stabilize, consistent with the sub-linear confidence scaling described above.

**Routing cascade efficiency.** In installations with mature personal learning data (100+ messages), the learned router handles approximately 60-70% of messages at cascade levels 1-3 (exact match, Bayesian, keyword) without any LLM classifier call. The collective layer covers an additional fraction for users who have opted in, reducing the LLM fallback rate further.

**Cold start mitigation.** New installations that opt into collective learning immediately benefit from routing hints and tool sequence patterns, bypassing the cold start period that local-only installations must endure. This is the most directly user-visible benefit of the collective layer.

## 10. Limitations

### 10.1 Cold Start

The collective system requires a critical mass of signals to produce reliable recommendations. For rare task categories or novel tool combinations, collective data may be insufficient, and the system falls back to LLM classification. This is mitigated by the cascaded design — the system gracefully degrades to local learning and then to LLM routing when collective data is sparse.

### 10.2 Signal Quality

The quality of collective recommendations depends on the quality of individual signals. If many installations produce low-quality routing decisions (e.g., due to misconfigured agents or adversarial input), the aggregated recommendations degrade. The confidence weighting (which down-weights low-confidence, low-quality signals) provides partial protection, but it is not a complete defense.

### 10.3 Adversarial Poisoning

A determined adversary could create many kbot installations, opt into collective learning, and send crafted signals designed to bias routing recommendations toward suboptimal agents. The current system does not implement Byzantine fault tolerance. Defenses include: rate limiting per-signal source (not currently implemented, as signals carry no source identity), statistical outlier detection in the aggregation step, and the sample count minimum (50 signals) which raises the cost of poisoning an attack.

### 10.4 Category Granularity

The current task classification system recognizes 12 categories. Real developer tasks are far more diverse — "fix a race condition in a React useEffect hook that only manifests in StrictMode" is meaningfully different from "fix a typo in a README," yet both classify as "debugging." Finer-grained categories would improve recommendation quality but require more signals per category to reach confidence thresholds.

### 10.5 Evaluation Limitations

We lack controlled experimental data comparing installations with and without collective learning under matched conditions. The observations in Section 9 are from real deployments where many variables differ simultaneously. Rigorous evaluation would require A/B testing with randomized assignment, which we have not yet conducted.

## 11. Future Work

### 11.1 Differential Privacy

We plan to add formal differential privacy guarantees using the Gaussian mechanism. Each numeric field in a signal (confidence, quality, message length) would be perturbed with calibrated noise before transmission, ensuring that the presence or absence of any individual signal changes the aggregate by at most epsilon. This would provide a mathematically rigorous privacy guarantee beyond the current hash-and-strip approach.

### 11.2 Federated Model Updates

The current collective system shares execution metadata, not model parameters. A natural extension is to share aggregated routing model updates — e.g., learned keyword-to-agent weights — using a federated averaging protocol. Each installation would periodically upload a delta to its local routing weights, and the server would compute a weighted average. This would allow the keyword voting system (cascade level 3) to improve collectively, not just individually.

### 11.3 Cross-Project Learning

Currently, project memory is local and per-directory. A collective extension could share anonymized project archetypes — "projects with React + TypeScript + Supabase tend to benefit from these tool sequences for debugging" — without revealing specific project details. This would allow kbot to recognize project types it has never seen locally but that resemble projects in the collective.

### 11.4 Temporal Signal Modeling

The current aggregation treats all signals within the 30-day window as exchangeable. A more sophisticated model would capture temporal dynamics — e.g., routing recommendations that were valid for GPT-4 may not apply to GPT-5. Time-series modeling of signal quality could detect model transitions and automatically re-weight recommendations.

### 11.5 Hierarchical Stigmergy

Grasse's original stigmergy model [Grasse, 1959] described a single shared medium. We envision a hierarchical extension: team-level stigmergy (signals shared within an organization), community-level stigmergy (the current global pool), and ecosystem-level stigmergy (cross-tool signals shared between kbot and other AI agents that adopt the same signal protocol). Each level would have different privacy boundaries and aggregation rules.

### 11.6 Adversarial Robustness

We plan to implement a robust aggregation protocol that can tolerate up to `f` adversarial signals out of `N` total using trimmed mean or coordinate-wise median aggregation [Yin et al., 2018]. This would provide formal guarantees against the poisoning attacks described in Section 10.3.

## 12. Conclusion

We have described a federated stigmergic learning system that is deployed in a production AI agent used by thousands of developers. The system is layered: personal learning caches patterns, solutions, and user profiles locally; Bayesian skill ratings provide probabilistic agent routing that improves with every interaction; and a collective intelligence layer aggregates anonymized execution signals across all participating installations into routing hints and tool sequence recommendations.

The key design decisions are: (a) share execution metadata, not code or content; (b) use SHA-256 hashing and field sanitization to guarantee that no personally identifiable or commercially sensitive information leaves the local machine; (c) require opt-in consent; (d) set confidence thresholds and sample count minimums that prevent premature convergence on low-quality recommendations; and (e) use asymmetric confidence thresholds that trust local history more than community data.

The system is stigmergic in the formal sense: agents coordinate indirectly through a shared medium (the signal database) without any direct agent-to-agent communication. It is federated in the sense that learning happens locally and only anonymized summaries are shared. And it exhibits network effects: each contributing installation makes the system more valuable for all others.

This work suggests that the stateless paradigm for AI developer tools is not a fundamental constraint but an engineering choice — one that can be replaced with a learning architecture that respects privacy, improves over time, and compounds across a community of users. The code is open-source, the protocol is documented, and the system is shipping.
